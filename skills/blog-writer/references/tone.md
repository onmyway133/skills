# Tone and Voice

## Overview

Technical writing should be accessible without being casual, professional without being cold.

## The Accessible Professional Voice

### Conversational Elements

Humanize content with natural phrases:

```markdown
# Good
"The good news, though, is that this can be fixed easily."
"Alright, just one piece of the puzzle left."
"Sound familiar?"

# Too formal
"It should be noted that this issue is resolvable."
"The final component will now be addressed."
```

### Collaborative Language

Use "we" to position readers as partners:

```markdown
# Good
"Let's build a component that..."
"We can solve this by..."
"As we've seen..."

# Distancing
"The developer should..."
"One might consider..."
```

### Direct Address

Speak to the reader:

```markdown
# Good
"In your own projects, you would..."
"You'll notice that..."

# Impersonal
"Users of this approach will..."
```

## Transition Phrases

### Cause and Effect

```markdown
"One mistake that's sometimes made is... Instead, we should..."
"Each time you call this method, it triggers..."
"Because of this limitation, we need to..."
```

### Building

```markdown
"Building on that foundation..."
"With that in place, we can now..."
"Taking this further..."
```

### Acknowledging Complexity

```markdown
"While this adds some complexity, it pays off when..."
"This might seem like overkill for small apps, but..."
"Bear in mind that..."
```

### Pivoting

```markdown
"The good news, though..."
"That said..."
"On the other hand..."
```

## Emphasis and Importance

### Strategic Bold/Italics

```markdown
# For key terms
"This is where *composition* becomes essential."

# For warnings
"**Important:** Never call this on the main thread."
```

### Numbered Lists for Sequences

```markdown
"That does three important things:
1. First mechanism
2. Second mechanism
3. Performance benefit"
```

## Avoiding Common Pitfalls

### Over-Qualification

```markdown
# Avoid
"It's arguably somewhat important to perhaps consider..."

# Better
"Consider using..."
```

### False Excitement

```markdown
# Avoid
"This amazing technique will revolutionize your code!"

# Better
"This pattern simplifies complex state management."
```

### Assumed Knowledge

```markdown
# Avoid
"Obviously, you'll need to conform to Sendable."

# Better
"For thread safety, we need to conform to Sendable."
```

## Tone Calibration

| Article Type | Tone |
|--------------|------|
| Quick tip | Concise, direct |
| Tutorial | Friendly, guiding |
| Deep dive | Measured, thorough |
| Opinion piece | Conversational, personal |

## Voice Checklist

- [ ] Uses "we" and "you" appropriately
- [ ] Includes humanizing phrases
- [ ] Avoids unnecessary jargon
- [ ] Explains acronyms on first use
- [ ] Maintains consistent formality level
- [ ] Reads naturally when spoken aloud
